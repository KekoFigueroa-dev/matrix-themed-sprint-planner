# V2 implementation specification ‚Äî matrix-themed sprint planner

**Repo:** https://github.com/KekoFigueroa-dev/matrix-themed-sprint-planner  
**Goal:** Portfolio-grade product (engineers / recruiters / collaborators).  
**Aesthetic:** [stripe.dev](https://stripe.dev/)-level polish + **restrained** retro cyberpunk / vaporwave (readable contrast, no neon soup).

**Platform (non-negotiable):**

- Real **email + password** auth (Supabase Auth).
- Real **multi-user** collaboration (shared workspace).
- **Supabase Postgres + RLS** ‚Äî RLS is the enforcement layer; UI gating is UX only.
- **No custom backend application server** in V2.
- **Deploy on Vercel.**
- **One logical workspace per primary owner flow** (see ‚ÄúWorkspace model‚Äù): first login **auto-creates** workspace; user is **owner** and acts as **admin** for permissions.
- **Admin** invites collaborators by email (**no invite expiry** in V2).
- **Permissions:** members **CRUD tasks**; **only admins** manage **projects / sprints / invites** (and membership changes per RLS).
- **Scope:** tight ‚Äî **not** a Jira clone.

**Companion docs:** [RLS policy model](./rls.md) ¬∑ [Agent workflow / progress pointer](../AGENTS.md) ¬∑ [README setup & deploy](../README.md)

### Table sketches (Postgres)

**A) `workspaces`**

| Column       | Type        | Notes |
|-------------|-------------|--------|
| id          | uuid PK     | `default gen_random_uuid()` |
| owner_id    | uuid NOT NULL | `references auth.users(id)` |
| name        | text NOT NULL | |
| created_at  | timestamptz | `default now()` |

**B) `workspace_members`**

| Column        | Type |
|---------------|------|
| workspace_id  | uuid `references workspaces(id) on delete cascade` |
| user_id       | uuid `references auth.users(id) on delete cascade` |
| role          | text NOT NULL `check (role in ('admin','member'))` |
| created_at    | timestamptz `default now()` |
| **PK**        | `(workspace_id, user_id)` |

Owner should have a row with `role = 'admin'` (or equivalent) so permission checks stay uniform.

**C) `invites`**

| Column        | Type |
|---------------|------|
| id            | uuid PK |
| workspace_id  | uuid `references workspaces(id) on delete cascade` |
| email         | text NOT NULL |
| role          | text NOT NULL `check (role in ('admin','member'))` default `'member'` |
| invited_by    | uuid `references auth.users(id)` |
| created_at    | timestamptz `default now()` |

No expiry in V2. On accept: delete row or mark accepted (implementation choice).

**D) `projects`**

| Column        | Type |
|---------------|------|
| id            | uuid PK |
| workspace_id  | uuid NOT NULL `references workspaces(id) on delete cascade` |
| name          | text NOT NULL |
| created_at    | timestamptz `default now()` |

**E) `sprints`**

| Column        | Type |
|---------------|------|
| id            | uuid PK |
| workspace_id  | uuid NOT NULL `references workspaces(id) on delete cascade` |
| project_id    | uuid NULL `references projects(id) on delete set null` |
| name          | text NOT NULL |
| start_date    | date NULL |
| end_date      | date NULL |
| created_at    | timestamptz `default now()` |

**F) `tasks`**

| Column        | Type |
|---------------|------|
| id            | uuid PK |
| workspace_id  | uuid NOT NULL `references workspaces(id) on delete cascade` |
| project_id    | uuid NULL `references projects(id) on delete set null` |
| sprint_id     | uuid NULL `references sprints(id) on delete set null` |
| title         | text NOT NULL |
| status        | text NOT NULL ‚Äî `'todo'` \| `'doing'` \| `'done'` |
| created_by    | uuid NULL `references auth.users(id)` |
| created_at    | timestamptz `default now()` |
| updated_at    | timestamptz `default now()` (maintain via trigger) |

## Role / permission matrix (product + RLS)

| Capability | Member (`workspace_members.role = 'member'`) | Admin (`role = 'admin'`) |
|------------|-----------------------------------------------|---------------------------|
| Read workspace-scoped data they belong to | Yes | Yes |
| CRUD **tasks** | Yes | Yes |
| Create / update / delete **projects** | No | Yes |
| Create / update / delete **sprints** | No | Yes |
| Create / read / revoke **invites** | No | Yes |
| Add/remove **workspace_members** (except controlled self-join path) | No | Yes (per policies) |

**Non-members:** no row visibility (RLS returns nothing).

**Invite accept:** implemented so a user with matching verified email can join **without** opening broad insert on `workspace_members` to everyone ‚Äî prefer **security definer RPC** (see Phase 3).

---

## Critical flows

1. **First login / sign-up:** User authenticates ‚Üí app calls **`ensure_workspace_for_user`** (or equivalent) ‚Üí one workspace, one `workspace_members` admin row ‚Üí no duplicate on refresh (idempotent).
2. **Admin invites:** Admin creates `invites` row (email + role). Invitee signs up / logs in with that email ‚Üí sees pending invite ‚Üí **Accept** runs RPC ‚Üí membership inserted, invite removed or marked accepted.
3. **Task CRUD:** Member or admin in workspace creates/updates/deletes tasks; RLS restricts to `workspace_id` where user is a member.
4. **Admin structure:** Admin creates project (optional) and sprint; member selects sprint context and works tasks.

## Out of scope (guardrails ‚Äî do not creep)

- Custom Node/Express/Fastify **application** server (serverless Next.js API routes only if needed; prefer Supabase RPC for privileged steps).
- Full Jira / Linear parity (workflows, plugins, reporting suite, custom fields enterprise).
- Invite expiry (explicitly V2 **no** expiry).
- Non‚Äìemail-password primary auth (OAuth-only, etc.) ‚Äî optional follow-up only.

### Phase 1: Maintainer test plan (after applying SQL)

**You apply the migration** (Supabase Dashboard ‚Üí SQL Editor ‚Üí paste the file, or Supabase CLI linked to the project), then verify:

1. **Tables exist:** `workspaces`, `workspace_members`, `invites`, `projects`, `sprints`, `tasks` ‚Äî Database ‚Üí Tables; **RLS enabled** on each.
2. **Bootstrap path (two users):** create two auth users in Authentication. As **User A** (SQL or Supabase client with A‚Äôs JWT): insert one `workspaces` row with `owner_id = A` and `name`; insert `workspace_members` `(workspace_id, A, 'admin')`. Confirm second workspace for same `owner_id` fails (unique index).
3. **Member isolation:** as **User B** (no membership), `select * from tasks` / `projects` returns **no rows** from A‚Äôs workspace (use Table editor filtered by JWT, or client with B‚Äôs session).
4. **Member vs admin (use User A as admin, add User B as member via admin policy):** User A inserts `workspace_members` for B with role `member`. As **B**: can `insert/update/delete` on `tasks` for that `workspace_id`; **cannot** mutate `projects` / `sprints` / `invites` (expect RLS error or zero rows).
5. **Invite visibility:** User A inserts `invites` for B‚Äôs email. **A** sees row; **B** sees row when `auth.jwt()->>'email'` matches (Supabase email claim). Third user **C** does not see it.

**Note:** Phase 2 adds **`ensure_workspace_for_user`** (see migration `20250515120000_ensure_workspace_for_user.sql`). `accept_workspace_invite` remains Phase 3.
