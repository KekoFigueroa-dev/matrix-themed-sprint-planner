# V2 implementation specification — matrix-themed sprint planner

**Repo:** https://github.com/KekoFigueroa-dev/matrix-themed-sprint-planner  
**Goal:** Portfolio-grade product (engineers / recruiters / collaborators).  
**Aesthetic:** [stripe.dev](https://stripe.dev/)-level polish + **restrained** retro cyberpunk / vaporwave (readable contrast, no neon soup).

**Platform (non-negotiable):**

- Real **email + password** auth (Supabase Auth).
- Real **multi-user** collaboration (shared workspace).
- **Supabase Postgres + RLS** — RLS is the enforcement layer; UI gating is UX only.
- **No custom backend application server** in V2.
- **Deploy on Vercel.**
- **One logical workspace per primary owner flow** (see “Workspace model”): first login **auto-creates** workspace; user is **owner** and acts as **admin** for permissions.
- **Admin** invites collaborators by email (**no invite expiry** in V2).
- **Permissions:** members **CRUD tasks**; **only admins** manage **projects / sprints / invites** (and membership changes per RLS).
- **Scope:** tight — **not** a Jira clone.

**Companion docs:** [RLS policy model](./rls.md) · [Agent workflow / progress pointer](../AGENTS.md) · [README setup & deploy](../README.md)

**V2 status:** **Complete** (phases 0–5). **Live:** https://matrix-themed-sprint-planner.vercel.app

## Entities and relationships

```
auth.users
    │
    ├──< workspaces.owner_id
    │
    ├──< workspace_members (workspace_id, user_id, role)
    │
    ├──< invites.invited_by
    │
    └──< tasks.created_by (optional audit)

workspaces (1) ──< (N) workspace_members
workspaces (1) ──< (N) invites
workspaces (1) ──< (N) projects
workspaces (1) ──< (N) sprints
workspaces (1) ──< (N) tasks

projects (1) ──< (N) sprints [optional project_id on sprint]
projects (1) ──< (N) tasks [optional project_id]

sprints (1) ──< (N) tasks [optional sprint_id]
```

### Table sketches (Postgres)

**A) `workspaces`**

| Column       | Type        | Notes |
|-------------|-------------|--------|
| id          | uuid PK     | `default gen_random_uuid()` |
| owner_id    | uuid NOT NULL | `references auth.users(id)` |
| name        | text NOT NULL | |
| created_at  | timestamptz | `default now()` |

**B) `workspace_members`**

| Column        | Type |
|---------------|------|
| workspace_id  | uuid `references workspaces(id) on delete cascade` |
| user_id       | uuid `references auth.users(id) on delete cascade` |
| role          | text NOT NULL `check (role in ('admin','member'))` |
| created_at    | timestamptz `default now()` |
| **PK**        | `(workspace_id, user_id)` |

Owner should have a row with `role = 'admin'` (or equivalent) so permission checks stay uniform.

**C) `invites`**

| Column        | Type |
|---------------|------|
| id            | uuid PK |
| workspace_id  | uuid `references workspaces(id) on delete cascade` |
| email         | text NOT NULL |
| role          | text NOT NULL `check (role in ('admin','member'))` default `'member'` |
| invited_by    | uuid `references auth.users(id)` |
| created_at    | timestamptz `default now()` |

No expiry in V2. On accept: delete row or mark accepted (implementation choice).

**D) `projects`**

| Column        | Type |
|---------------|------|
| id            | uuid PK |
| workspace_id  | uuid NOT NULL `references workspaces(id) on delete cascade` |
| name          | text NOT NULL |
| created_at    | timestamptz `default now()` |

**E) `sprints`**

| Column        | Type |
|---------------|------|
| id            | uuid PK |
| workspace_id  | uuid NOT NULL `references workspaces(id) on delete cascade` |
| project_id    | uuid NULL `references projects(id) on delete set null` |
| name          | text NOT NULL |
| start_date    | date NULL |
| end_date      | date NULL |
| created_at    | timestamptz `default now()` |

**F) `tasks`**

| Column        | Type |
|---------------|------|
| id            | uuid PK |
| workspace_id  | uuid NOT NULL `references workspaces(id) on delete cascade` |
| project_id    | uuid NULL `references projects(id) on delete set null` |
| sprint_id     | uuid NULL `references sprints(id) on delete set null` |
| title         | text NOT NULL |
| status        | text NOT NULL — `'todo'` \| `'doing'` \| `'done'` |
| created_by    | uuid NULL `references auth.users(id)` |
| created_at    | timestamptz `default now()` |
| updated_at    | timestamptz `default now()` (maintain via trigger) |

## Role / permission matrix (product + RLS)

| Capability | Member (`workspace_members.role = 'member'`) | Admin (`role = 'admin'`) |
|------------|-----------------------------------------------|---------------------------|
| Read workspace-scoped data they belong to | Yes | Yes |
| CRUD **tasks** | Yes | Yes |
| Create / update / delete **projects** | No | Yes |
| Create / update / delete **sprints** | No | Yes |
| Create / read / revoke **invites** | No | Yes |
| Add/remove **workspace_members** (except controlled self-join path) | No | Yes (per policies) |

**Non-members:** no row visibility (RLS returns nothing).

**Invite accept:** implemented so a user with matching verified email can join **without** opening broad insert on `workspace_members` to everyone — prefer **security definer RPC** (see Phase 3).

---

## Critical flows

1. **First login / sign-up:** User authenticates → app calls **`ensure_workspace_for_user`** (or equivalent) → one workspace, one `workspace_members` admin row → no duplicate on refresh (idempotent).
2. **Admin invites:** Admin creates `invites` row (email + role). **No invite email is sent by the app.** Invitee signs up / logs in with **that exact email** → `/invites` → **Accept** runs RPC → membership inserted, invite removed.
3. **Task CRUD:** Member or admin in workspace creates/updates/deletes tasks; RLS restricts to `workspace_id` where user is a member.
4. **Admin structure:** Admin creates project (optional) and sprint; member selects sprint context and works tasks.

### Phase 1: Maintainer test plan (after applying SQL)

**You apply the migration** (Supabase Dashboard → SQL Editor → paste the file, or Supabase CLI linked to the project), then verify:

1. **Tables exist:** `workspaces`, `workspace_members`, `invites`, `projects`, `sprints`, `tasks` — Database → Tables; **RLS enabled** on each.
2. **Bootstrap path (two users):** create two auth users in Authentication. As **User A** (SQL or Supabase client with A’s JWT): insert one `workspaces` row with `owner_id = A` and `name`; insert `workspace_members` `(workspace_id, A, 'admin')`. Confirm second workspace for same `owner_id` fails (unique index).
3. **Member isolation:** as **User B** (no membership), `select * from tasks` / `projects` returns **no rows** from A’s workspace (use Table editor filtered by JWT, or client with B’s session).
4. **Member vs admin (use User A as admin, add User B as member via admin policy):** User A inserts `workspace_members` for B with role `member`. As **B**: can `insert/update/delete` on `tasks` for that `workspace_id`; **cannot** mutate `projects` / `sprints` / `invites` (expect RLS error or zero rows).
5. **Invite visibility:** User A inserts `invites` for B’s email. **A** sees row; **B** sees row when `auth.jwt()->>'email'` matches (Supabase email claim). Third user **C** does not see it.

**Note:** Phase 2 adds **`ensure_workspace_for_user`** (see migration `20250515120000_ensure_workspace_for_user.sql`). `accept_workspace_invite` remains Phase 3.
